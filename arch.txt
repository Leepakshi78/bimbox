When a user registers, the backend first validates the email and password,
 hashes the password for security, and creates the user with a verification 
 status set to false. It then generates a one-time password (OTP) 
 with a short expiry time and sends it to the user’s email. 
 The user must enter this OTP to verify their email, 
 after which the backend marks the account as verified 
 and removes the OTP. Only verified users are allowed to log in.
 During login, the backend checks the credentials and issues a JWT token,
 which the frontend uses to access protected routes.

If a user forgets their password, 
the backend again generates an OTP and emails
it to the user for identity confirmation. Once the OTP is verified,
the user is allowed to set a new password, which is hashed and saved 
securely in the database. This ensures that email ownership is
confirmed both during registration and password recovery,
while JWT handles secure login sessions afterward.

 We use Pino instead of Winston because Pino is faster, lightweight, and produces
  structured logs by default, making it better for high-performance Node.js applications.
   Winston is more flexible but heavier and slower for large-scale logging.


utils/logger.js: creates one common logger (Pino). This is the main character that prints structured logs to terminal.


middlewares/httpLogger.js: logs every request automatically (method, URL, status code). So you can track which API was hit and what response status it gave.

middlewares/errorLogger.js: logs every error (message + stack + route info). This helps you debug without relying only on the client response.

middlewares/error.js (global error handler): sends a clean JSON response to the client when any error happens, so your app doesn’t crash and responses stay consistent.

Services/Controllers can also log important events (example: “OTP sent”) using the same logger.
